# M9c — Persistent Fog + GM Vision Mode: Plan

> **For agentic workers:** execute with **`mainline-plan-execution`** (inline enumerative
> per-task spec check + ONE dispatched final branch review; **buddy-check it** — security-sensitive:
> explored fog is per-player secret memory, and the GM see-as-player path crosses recipients).
> Checkbox steps. **Read M9 spec §7 + §10 D-V2/D-V3 and the M9b "Implementation deviations" note
> before executing.**

**Goal:** Close M9. Persistent, server-authoritative, per-(scene,player) **explored** memory
accumulated from each vision recompute and dispatched alongside the live `visible` polygons; an
engine-owned **three-state** fog shader (unexplored = black, explored-but-not-visible = dimmed,
visible = clear); and a **client-side** GM vision toggle (see-all / see-as-player).

**Spec:** [`../specs/2026-06-22-m9-walls-vision-fog-design.md`](../specs/2026-06-22-m9-walls-vision-fog-design.md)
§7 (fog), §10 D-V2/D-V3. Builds on M9b (`vision` channel, `player_vision_polygons`, two-state fog,
`compute_derived`) + the M8a `SceneDerived` dispatch in `ws/conn.rs`.

## Design decisions (spec-locked + implementation forks resolved; flag forks for review)

Spec §7 locks the user-owned decisions: persistent per-(scene,player) server storage; storage shape
= **sparse explored-cell set default**, polygon-union opt-in, GM-configurable per scene; three-state
shader; GM toggle is client-side ("no extra server path"). These implementation forks were resolved
to ship M9c; **flag for review**:

1. **Explored accumulation lives in the dispatch layer, not `compute_derived`.** `compute_derived`
   stays pure/sync over the ECS (returns the live `visible` polygons + their cells). Accumulation
   (read stored explored → union the new visible cells → persist-if-grown → merge `explored` into
   the payload) needs async DB access, so it runs in `ws/conn.rs` at the two dispatch sites
   (initial `SceneSubscribe`, coalesced re-dispatch). Rationale: keeps the raycaster headless-pure
   and testable; the DB-touching async accumulation is a dispatch concern.
2. **Explored storage = a dedicated table `explored_fog(world_id, scene_id, user_id, cells)`**, NOT
   a document. Documents broadcast to all members; explored is per-recipient SECRET memory. `cells`
   is a serialized **sparse set of packed grid-cell coords** (the §7 default). Keyed
   `(scene_id, user_id)`. The polygon-union opt-in (§7) is **deferred to a follow-up** — the table
   + shader + dispatch ship the cell-set mode first (the default); the union mode is a per-scene GM
   setting added later. (Decompose, don't descope: the union mode is a representation swap behind
   the same `explored` payload + shader, addable without rework.)
3. **Cell quantization = the scene's `system.grid.size`.** Cell `(i,j) = (floor(x/size),
   floor(y/size))`. A vision recompute marks every cell whose **center** lies inside any `visible`
   polygon (bounded, resolution = grid cell — fine for the dimmed memory layer per §7).
4. **Dispatch encoding:** the `vision` payload gains `explored: { cell: <size>, cells: [i,j,…] }`
   (flat coord pairs, mirroring `points`). The client builds cell rects → the `explored` mask. The
   GM payload (`mode:"all"`) carries no `explored` (GM has no fog). Empty/garbled `explored` →
   treated as no explored memory (fail-safe: less reveal, never more).
5. **GM see-as-player decomposed to M9c-2 — Approach B (on-demand, user-confirmed 2026-06-22).**
   M9c-1 ships the GM **see-all** toggle (client-only). **See-as-player** = a `SceneSubscribe`
   parameter (`as_user`) the server honors **only for GMs**: it computes the `vision` payload as the
   target player (their masked polygons + their explored) and returns ONLY that player's masks.
   **Chosen over the spec §7/§10 "no extra server path" variant (Approach A: ship *all* players'
   masks to the GM, client selects)** on the user's durable/secure/performant directive:
   - **Secure (least-privilege):** the GM receives only the one player they actively view, on
     demand, behind a single explicit GM-auth gate — not every player's secret state continuously
     in flight. **Authorization is the security-critical surface: a non-GM `as_user` MUST be
     rejected, and the GM impersonates only via a server-resolved target context (never client-
     supplied role/scope).**
   - **Performant:** O(1 player) computed on demand vs O(all players × explored) on every GM
     dispatch even when not viewing-as-anyone.
   - **Durable:** an explicit, minimal, auditable wire contract vs an ever-growing GM payload.
   This is a deliberate **deviation from spec §10 D-V3's "no extra server path"** note; the spec is
   updated to record it. Buddy-check the cross-recipient authorization.

## Decomposition

- **M9c-1 — Persistent explored fog + three-state shader + GM see-all toggle.** The table +
  migration; the explored-store (async accumulate/persist); dispatch-layer accumulation + payload
  `explored`; the client `explored` mask + three-state shader; the GM see-all client toggle.
- **M9c-2 — GM see-as-player.** The `SceneSubscribe { as_user }` parameter, GM-only authorization,
  server computes `compute_derived` as the target player (incl. that player's explored), client GM

## M9c-1 tasks

### Task 1: `explored_fog` table + migration + the explored-store (server, headless)
**Files:** `src/server/migrations/0007_explored_fog.sql` (new), `src/server/src/scene/explored.rs`
(new) or fold into `data/`, wired in the repo/AppState.
- Migration: `CREATE TABLE explored_fog (world_id, scene_id, user_id, cells BLOB NOT NULL,
  PRIMARY KEY (scene_id, user_id))`. `cells` = a serialized sparse cell set (sorted packed
  `i64 = (i as i32 as i64) << 32 | (j as u32 as i64)`; or a compact bincode/JSON of `Vec<(i32,i32)>`).
- `ExploredStore`: `load(scene, user) -> ExploredSet`, `accumulate(scene, user, new_cells) ->
  (ExploredSet, grew: bool)` (union; `grew` gates the DB write), `persist(scene, user, &set)`.
- `ExploredSet`: a `HashSet<(i32,i32)>` (or `BTreeSet` for deterministic serialization) with
  `mark_polygon(points, cell_size)` (center-in-poly rasterize, reusing the M9b point-in-poly idea).
- **Tests:** accumulate is monotone (revisiting marks no new cells → `grew=false`); a polygon marks
  the expected cells; round-trip serialize/deserialize; per-(scene,user) isolation.
- [ ] **Commit:** `feat(m9c-1): explored_fog table + explored-store (accumulate/persist)`

### Task 2: dispatch-layer accumulation + `explored` in the vision payload (server)
**Files:** `src/server/src/scene/mod.rs` (have `compute_derived` also return the visible cells, or
add a sibling that yields cells), `src/server/src/ws/conn.rs` (both dispatch sites).
- `compute_derived("vision")` for a player already returns scene-tagged polygons; add the per-scene
  visible **cells** (or compute them in the dispatch layer from the polygons + the scene grid size).
- In `conn.rs`, for a non-GM `vision` dispatch: for each scene's visible cells, `accumulate` into the
  store (persist if grown), then merge `explored: { cell, cells }` into the payload before sending.
  Keep the watermark = `committed_seq` semantics unchanged.
- **Concurrency:** the accumulate-then-persist must not race two of the same user's sockets; key the
  store write by `(scene,user)` and make `accumulate+persist` atomic (single await, last-writer-wins
  is safe since union is monotone). No write when `grew=false` (avoids churn on a still scene).
- **Tests (integration):** a player subscribes, moves, re-subscribes → `explored` grows and persists
  across a reconnect (new connection sees prior explored); a GM payload carries no `explored`.
- [ ] **Commit:** `feat(m9c-1): accumulate + dispatch persistent explored fog`

### Task 3: client three-state fog shader + `explored` mask (render)
**Files:** `src/client/render/src/types.ts` (`VisibilityInput.explored` already typed as
`Polygon[]`? — switch to a cell-rect list or keep `Polygon[]` built from cells), `engine.ts`
(`toVisibility` parses `explored`), `pixi-backend.ts` (`setVisibility` three-state), tests.
- `toVisibility`: parse `explored.cells` (+ `explored.cell` size) → rects (scene-filtered, same as
  `visible`); fail-safe (missing/garbled → no explored). Keep fail-CLOSED for `visible`.
- `setVisibility` three states via two stacked sheets in the `mask` layer:
  - **Sheet A** (darkest, ~0.92 black): cut by `explored ∪ visible` holes → unexplored stays black.
  - **Sheet B** (dim, ~0.45 black): cut by `visible` holes → explored shows dimmed, visible clear.
  - Net: unexplored = A+B; explored = B; visible = none. Tune alphas to the §7 three states.
- **Tests (render unit):** `explored` rects reach the explored hole-set; an `all` frame clears both;
  empty explored + empty visible = full dark fog; cross-scene explored is filtered out.
- [ ] **Commit:** `feat(m9c-1): three-state fog shader (unexplored/explored/visible)`

### Task 4: GM see-all client toggle (ui) + e2e
**Files:** a scene-tools or core-ui control (GM-only) toggling the engine's fog rendering;
`engine.ts` exposes a `setGmFog(on)` (when off, force `mode:"all"`); `stage.spec.ts` e2e.
- The GM gets `mode:"all"`; the toggle lets the GM render fog with an empty player-less mask (full
  fog preview) or no fog. Client-only; no server path (D-V3).
- **e2e:** as GM, toggling shows/hides fog; a player (token) sees three-state fog (visible clear,
  rest dark) — extend the M9b vision e2e with a `data-explored`/`data-fog-state` signal.
- [ ] **Commit:** `feat(m9c-1): GM see-all fog toggle + e2e`

## M9c-2 tasks (Approach B — on-demand, GM-authorized see-as-player)

### Task 1: wire `as_user` + server GM-authorized see-as-player dispatch (server, headless)
**Files:** `ws/protocol.rs` (`ClientMsg::SceneSubscribe` gains `as_user: Option<Uuid>`, default None),
`ws/conn.rs` (`Egress::SceneSubscribe` + `SceneSub` carry the effective target context; both dispatch
sites use it), ts-rs regen if the frame is exported.
- `scene_subscribe` gains optional `as_user`. On the egress `SceneSubscribe`:
  - `as_user == None` → today's behavior (the connection's own `ctx`).
  - `as_user == Some(target)` → **require `ctx.world_role == Gm`**; else reject with `SceneError`
    ("not authorized to view as another user") and do NOT register the subscription.
  - For a GM: resolve the target's role **server-side** via `repo.member_role(world, target)` (reject
    if the target is not a member) → build `effective_ctx = PermissionContext { user_id: target,
    world_role: <resolved> }`. NEVER trust a client-supplied role/scope. Store `effective_ctx` (or
    the resolved `(target, role)`) in `SceneSub`.
- Both `compute_derived` + `enrich_vision_explored` call sites use `effective_ctx` so the GM receives
  EXACTLY the target player's masked polygons + that player's explored (and the explored accumulation
  is attributed to the target, not the GM — read-only impersonation must not mutate the GM's own row;
  acceptable that viewing-as marks the target's explored, since it equals what the target would see).
  **Decide + document:** viewing-as should NOT grow the target's explored from the GM's session — gate
  the accumulation to the connection's own user (skip the persist when `as_user` is set; still emit the
  target's stored explored read-only). This keeps see-as a pure observer.
- **Tests (integration, using the M9c-1 `add_player` harness):** a GM subscribes vision `as_user=player`
  → receives `mode:"masked"` with the player's polygons + explored; a **non-GM** `as_user` → `SceneError`
  (the critical auth gate); `as_user` = a non-member → rejected; viewing-as does not persist explored
  under the GM's session.
- [ ] **Commit:** `feat(m9c-2): GM-authorized see-as-player vision dispatch (as_user)`

### Task 2: client `as_user` seam + GM player-picker (core + render + ui)
**Files:** `@shadowcat/core` `SubscribeScene` type + `worldSession` (send `as_user` in the frame),
`@shadowcat/render` `engine` (`setViewAsUser(id|null)` re-subscribes the vision channel), `AppContext`,
`Stage.svelte` (GM-only picker), `stage.spec.ts`.
- `SubscribeScene` gains an optional `{ asUser?: string }`; `worldSession.subscribeScene` forwards it
  into the `scene_subscribe` frame. `engine.setViewAsUser(userId | null)` tears down + re-establishes
  the `vision` subscription with `as_user` (null → the GM's own see-all).
- **GM picker (player-list source — resolve in this task):** prefer an existing members source; if none
  is exposed client-side, derive the candidate players from distinct **token `owner`s** the GM already
  sees (label best-effort) rather than adding a members API in this milestone. "See all" + one entry
  per candidate player → `setViewAsUser`.
- **e2e:** as GM, pick a player → `data-vision-mode` flips to `masked` (the GM now sees that player's
  view) in real GL; back to "See all" → `all`.
- [ ] **Commit:** `feat(m9c-2): client see-as-player seam + GM picker + e2e`
