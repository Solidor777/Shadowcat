# M8.5 · UI Packaging Decomposition — Design

> Mechanical extraction of today's monolithic `@shadowcat/ui` into the target
> client packaging from `design/ARCHITECTURE.md` §1: a thin app shell, a
> swappable entry package, and per-element in-game UI packages. **No behavior
> change** — pure refactor. Realizes a planned milestone (`PLAN.md` M8.5),
> sequenced after the M8 UI/render foundation and before M12 (default modules,
> which depends on per-element packaging).

## 1. Goal & non-goals

**Goal.** Resolve the browser client into independently replaceable layers so a
self-hoster can swap the entry experience (external auth/identity) and a modder
can replace any in-game element (region/panel/tool) in isolation — without
forking the whole UI.

**Non-goals.**
- Changing the contract/surface model itself (built in M6b/M7 — unchanged).
- Changing the dynamic external-module loader path (`ModuleRegistry`/`loadModules`
  for third-party runtime modules — unchanged; this milestone is about how
  *first-party default* UI is packaged).
- Any behavior, visual, or protocol change. Tests move with their code; the
  Playwright-vs-binary e2e stays green; the `dist/` → `rust-embed` path is
  unchanged.

## 2. Current state (verified 2026-06-24)

`@shadowcat/ui` (`src/client/ui`) wears three hats:

- **Shell** — `main.ts` (mount), `App.svelte` (boot + routing + world
  enter/leave orchestration), `lib/{route,worldSession,sessionState,appContext,
  api,i18n,reactiveStore,sceneInteraction}`, `lib/{Layout,Surface,Table}.svelte`.
- **Entry** — `lib/views/{Setup,Login,WorldSelect}.svelte`.
- **Content modules** (already seam-clean folders, not packages) —
  `modules/core-ui/*` (TopBar, Stage, StatusBar, Settings, Assets) and
  `modules/scene-tools/*` (ToolRail + tools).

**Coupling audit (load-bearing — confirms the split is mechanical):**
- No panel imports another panel; no tool imports a panel. All cross-element
  communication is through `getAppContext()`. The element-boundary discipline
  (ARCHITECTURE.md §1) held.
- Every module imports seams that physically live in shell `lib/` but are *not*
  shell internals: `appContext.ts`, `Surface.svelte`, `i18n.svelte.ts`, and the
  `sceneInteraction.ts` bridge. `appContext`/`Surface`/`i18n` use the Svelte
  runtime (`getContext`/`setContext`, `createSubscriber`), so they **cannot**
  live in `@shadowcat/core` (Svelte-free by invariant #7); `sceneInteraction` is
  framework-neutral (type-only `@shadowcat/render` imports) but is co-located
  because the `AppContext` type references it. None can stay in the shell — the
  shell imports the modules, so a module importing the shell would cycle.
- `App.svelte` statically imports `coreUi` + `sceneTools` and hands them to
  `WorldSession` directly. First-party modules are bundled-at-build-time
  workspace packages, **not** dynamically loaded — the model M8.5 formalizes.
- Two shell-internal imports inside content that must reroute through seams:
  `Settings.svelte` imports `logout` (from `lib/api`) and `navigate` (from
  `lib/route`). Everything else a module imports is already a seam or a
  cross-package dependency (`@shadowcat/{core,render,types}`).

## 3. Target package layering

A new shared Svelte UI-runtime package is the enabler. Final layering:

```
@shadowcat/core      Svelte-free headless: store, ws-client, modules,         [exists]
                     contributions, hooks, services, optimistic, assets.
@shadowcat/render    Engine-owned PixiJS render layer.                        [exists]
@shadowcat/ui-kit    NEW. Shared Svelte runtime binding the seams every Svelte
                     consumer needs. Depends on core (+ render types).
@shadowcat/shell     Thin app: bootstrap, routing, session, AppContext
                     provider, default-module-set wiring, locale boot. Depends
                     on ui-kit, core, and the default module packages.
src/modules/*        Default first-party packages, each depending on ui-kit
                     (+ core/render/types as needed):
                       @shadowcat/module-entry       (swappable; login + world mgmt)
                       @shadowcat/module-core-ui      (layout: grid + region surfaces)
                       @shadowcat/module-topbar
                       @shadowcat/module-statusbar
                       @shadowcat/module-stage        (canvas host panel)
                       @shadowcat/module-settings
                       @shadowcat/module-assets
                       @shadowcat/module-scene-tools  (relocated; already clean)
```

**Dependency direction:** `shell → modules → ui-kit → core`. No cycles. `ui-kit`
also takes a **type-only** `@shadowcat/render` dependency (`sceneInteraction`
references `SceneTool`/`SceneToolHost`/`Point`/`ShapeNodeSpec`); `AppContext`'s
`SceneFrame`/`SceneSubscription`/`WireOperation` are core types. `shell` and the
stage/scene-tools modules take a runtime `@shadowcat/render` dependency.

**Build model.** Default modules are workspace packages that the shell imports
statically; Vite tree-builds them into the one bundle that `rust-embed` bakes
into the binary. This is distinct from the dynamic `loadModules` path for
untrusted third-party modules (unchanged). "Replaceable" here means a
self-hoster forks/swaps a package and rebuilds — not runtime hot-swap.

### 3.1 `@shadowcat/ui-kit` contents

Moved verbatim from shell `lib/` (no logic change):
- `appContext.ts` — the `AppContext` interface, `TFunc`, `get/setAppContext`,
  `__APP_CONTEXT_KEY__`.
- `Surface.svelte` — the contribution host adapter.
- `i18n.svelte.ts` — the `t`/`locale`/`i18n` reactive adapter (the
  framework-neutral I18n core stays in `@shadowcat/core`; this is its Svelte
  binding).
- `sceneInteraction.ts` — the `SceneInteraction` interface + `SceneInteractionBridge`
  (referenced by the `AppContext` type; framework-neutral but co-located with
  AppContext for a single import origin).
- Test fixtures that seed an AppContext (`__fixtures__/appContextTest.ts`,
  `Probe`, `SurfaceHarness`, `I18nProbe`) move here as the kit's test support so
  every module can reuse them.

`reactiveStore.svelte.ts` stays where it is used; moves to ui-kit only if a
module imports it (verified during M8.5a — panels do not today).

### 3.2 `@shadowcat/shell` contents (the renamed, slimmed `@shadowcat/ui`)

Retains only app-level orchestration:
- `main.ts` (mount), the Vite app entry — `dist/` output path unchanged.
- `App.svelte` — boot sequence + route gating + world enter/leave + last-world
  auto-enter. Renders `<Entry {onEnterWorld}/>` pre-world; `<Table>` in-world.
- `lib/{route,worldSession,sessionState}.svelte.ts` — routing + session/world
  lifecycle.
- `lib/Table.svelte` — the in-world AppContext provider; renders the root surface.
- Session-lifecycle REST: `getMe`, `getUiState`/`putUiState`, `listWorldMembers`
  (worldSession uses it for see-as labels). The 6-line `getJson`/`postJson`
  helper is shared — duplicated into the two REST owners (shell, entry) and the
  assets module rather than introducing an HTTP package (a private 6-line
  function is below the bar for a shared dependency).
- Assembles the **default module set** array and the default `<Entry>`, passing
  them down. Swapping the entry or a module = editing this assembly + the
  package.json dep.

## 4. The thin shell ↔ content boundaries

### 4.1 Root surface + module-owned layout

`Table.svelte` (shell) sets `AppContext` and renders a single
`<Surface contract="shadowcat.surface:root"/>`. The `core-ui` module contributes
`Layout.svelte` (the responsive CSS grid + the five region `<Surface>`s) into
`root`, and declares `provides` for all six contracts (`root` + the five
regions: topbar/toolrail/stage/sidebar/statusbar). Panels/tools `require` and
contribute into the regions exactly as today.

Consequence: the entire layout (grid, region set, breakpoints) is swappable by
replacing one module — the shell hard-codes no region. This satisfies
ARCHITECTURE.md invariant #7 ("a module may replace … the application shell").

`core-ui`'s manifest gains `provides: [{root, singleton}, {topbar, singleton},
{stage, singleton}, {statusbar, singleton}, {toolrail, multi}, {sidebar, multi}]`
and contributes one component (`Layout` → `root`). Panel modules each `require`
their region contract and contribute their component (matching the current
`scene-tools` pattern).

### 4.2 Entry package contract

`@shadowcat/module-entry` exports a single Svelte component:

```ts
// <Entry> props
interface EntryProps {
  /** Called when the user selects a world to enter. The shell owns the
   *  session/WS lifecycle from here. */
  onEnterWorld: (worldId: string) => void;
}
```

`<Entry>` owns its internal `setup → login → world-select` flow and its REST
(`getConfig`, `login`, `setup`, `listWorlds`, `createWorld`, and future
world-management endpoints). It does **not** own the session, the WS connection,
or last-world auto-enter — those stay in the shell (a self-hoster replacing
`<Entry>` for external auth keeps the shell's session machinery intact).

**Boot division of labor.** The shell's `App.svelte` keeps the boot sequence:
on load it resolves `getMe` + last-world (a session concern), and if an
authenticated session with a valid last world exists it enters directly
(skipping entry). Otherwise it renders `<Entry {onEnterWorld}/>`. `<Entry>`
internally decides setup vs login vs world-select via `getConfig`/its own auth
state. The shell's `route` still distinguishes pre-world from in-world; entry's
internal step is entry-local state (it no longer needs the shell `route`).

Locale boot stays in the shell (`sessionState.loadSessionState` applies the
saved locale before either entry or in-world renders), so a swapped entry
inherits i18n for free.

### 4.3 AppContext session actions

`AppContext` gains `logout(): Promise<void>` alongside the existing
`leaveWorld()`. `Settings.svelte` calls `ctx.leaveWorld()` + `ctx.logout()`
instead of importing shell `lib/{api,route}`. The shell implements `logout`
(POST `/api/logout` + navigate to the pre-world route via its own router). No
other AppContext change.

### 4.4 WorldSession API

`WorldSessionOpts` changes from `{ coreUiModule: Module; featureModules?:
Module[] }` to `{ modules: Module[] }` — an ordered list the shell assembles
(layout/core-ui first so its region surfaces exist before panels activate;
activation order is cosmetic for rendering since `<Surface>` is reactive, but
kept deterministic). `#onWelcome` adds every module in order, then activates.
`reconcileTopology` is unchanged (all `provides` present before `activate`).

## 5. Sequencing — three dependency-ordered slices

Each slice is a self-contained refactor with green tests + a buddy-check, and
ships zero behavior change. Ordered by dependency.

**M8.5a — `ui-kit` + thin shell.**
Create `@shadowcat/ui-kit`; move `appContext`, `Surface.svelte`, `i18n.svelte`,
`sceneInteraction`, and the AppContext test fixtures into it. Rename
`@shadowcat/ui` → `@shadowcat/shell` (update `pnpm --filter` in CI, the
workspace, and `package.json`; `dist/` path unchanged). Repoint every existing
import (`../../../lib/appContext` → `@shadowcat/ui-kit`, etc.). core-ui and
scene-tools remain folders inside the shell for now (still compiling, now via
ui-kit). Establishes the layering everything else builds on.

**M8.5b — swappable entry package.**
Create `@shadowcat/module-entry`; move `lib/views/*` into it; define the
`<Entry>` callback contract; co-locate entry's REST (a private `entryApi`).
Shell's `App.svelte` renders `<Entry {onEnterWorld}/>`; the shell sheds the
entry REST and the entry view imports. Validates the swap boundary.

**M8.5c — per-element split + scene-tools relocation.**
Split core-ui into `@shadowcat/module-{core-ui(layout),topbar,statusbar,stage,
settings,assets}`. Introduce the `root` surface + module-owned `Layout`
(§4.1). Reroute Settings' `logout`/`navigate` through AppContext session
actions (§4.3). Co-locate Assets' asset-CRUD REST. Relocate `scene-tools`
(`src/client/ui/src/modules/scene-tools` → `src/modules/scene-tools`) as a
package — already seam-clean. Switch `WorldSession` to `modules: Module[]`
(§4.4) and the shell to assembling the default set.

## 6. Testing

- Unit/component tests (Vitest + `@testing-library/svelte`) move with their
  components into each package; the shared AppContext fixtures live in `ui-kit`
  and are imported by module test suites.
- Each package builds independently (`pnpm --filter <pkg> build` / `test`);
  CI's existing per-workspace test run covers them via the `src/*` globs.
- The Playwright smoke test (UI served by the binary) is unchanged and must stay
  green after each slice — the proof the refactor preserved behavior.
- `rust-embed` compile-ordering is unchanged: the shell still builds to `dist/`;
  CI still builds the client before `cargo` (rust-embed validates `../../dist/`
  at compile time). Only the build *entry package name* changes
  (`@shadowcat/ui` → `@shadowcat/shell`).

## 7. Risks & mitigations

- **Rename churn (`@shadowcat/ui` → `@shadowcat/shell`).** Touches CI `--filter`,
  `pnpm-workspace.yaml` (already globs `src/client/*`), and the Vite/embed wiring.
  Mitigation: M8.5a is exactly this rename + the ui-kit move, verified by a green
  binary-served Playwright run before proceeding.
- **Import-path sprawl.** Many `../../../lib/X` → `@shadowcat/ui-kit` rewrites.
  Mechanical; caught by `tsc`/Vite build failing loudly on a missed path.
- **Package proliferation (8 module packages + ui-kit).** Accepted: it is the
  stated target (per-element replaceability). Each is small and own-tested.
- **Root-surface indirection (§4.1).** One extra contract + a nested `<Surface>`.
  Net simplification: the shell stops hard-coding the region set.

## 8. Decisions (user-owned, confirmed 2026-06-24)

- Granularity: **fine-grained per-element** packages.
- REST: **co-located** with the feature that calls it; entry is **one swappable
  `<Entry>`** component with an `onEnterWorld` callback contract.
- Layout: **module-owned** (core-ui contributes `Layout` into a `root` surface);
  the shell is genuinely thin.
- Naming: `@shadowcat/{core,render,ui-kit,shell}` infra; `@shadowcat/module-*`
  default modules.
- Sequencing: **M8.5a → M8.5b → M8.5c**, each buddy-checked, zero behavior change.

See `design/ARCHITECTURE.md` §1 (target packaging + element-boundary discipline)
and `PLAN.md` M8.5.
